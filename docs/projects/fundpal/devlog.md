# 开发历程

记录 FundPal 从 idea 到实现的过程，包括关键决策、踩过的坑和阶段性成果。

## 📅 开发时间线

### 2024 Q4 — 项目启动

**起因**

作为一个每月坚持定投的开发者，我受够了各种基金 App 的广告轰炸和信息噪音。我只想要一个干净的工具，帮我：
- 用数据筛选基金，而不是看排行榜
- 管理投资组合，而不是零散地记在 Excel 里
- 遵守投资纪律，而不是被情绪左右

于是，FundPal 诞生了。

**初期技术选型**

- 前端选了 Vue 3 + TypeScript，因为这是我最熟悉的技术栈
- 后端先用 Node.js + Express，追求快速出原型
- 数据库选 SQLite，个人项目不需要 MySQL 那么重

### 2025 Q1 — 基金筛选 & 组合管理

**核心功能落地**

这个阶段集中精力把两个核心功能做出来：

1. **基金筛选器**
   - 接入了天天基金的公开数据
   - 实现了多维度筛选和排序
   - 踩坑：数据源不稳定，做了缓存和降级策略

2. **组合管理**
   - 支持创建多个组合
   - 实时计算收益和回撤
   - 用 ECharts 做了可视化看板

**关键踩坑**

> 基金净值数据获取比想象中复杂得多。不同数据源的更新时间不同、格式不统一，必须做好数据清洗和异常处理。

### 2025 Q2 — AI 能力集成（进行中）

**当前方向**

- 集成 OpenAI API，做基金分析报告
- 尝试用 RAG 方案，让 AI 能基于最新的基金公告做分析
- 探索本地模型方案（ollama + qwen），降低 API 成本

## 🧠 关键技术决策

### 为什么选 SQLite 而不是 MySQL

| 对比项 | SQLite | MySQL |
|--------|--------|-------|
| 部署成本 | 零配置 | 需要安装和维护 |
| 性能 | 读密集场景够用 | 更强的并发能力 |
| 适用场景 | 个人/小团队 | 企业级应用 |
| 数据量 | < 100万条够用 | 无上限 |

个人项目，SQLite 完全够用。未来如果要做多人版本，再迁移到 PostgreSQL。

### 前端状态管理方案

最终选了 Pinia 而不是 Vuex：
- API 更简洁，不需要 mutations
- TypeScript 支持更好
- 体积更小

### 数据缓存设计

投资数据有个特点：**历史数据不变，只有今天的数据在更新**。基于此设计了三级缓存：

```
请求 → 内存缓存（5min）→ SQLite（每日更新）→ 外部 API
```

这样既保证了数据新鲜度，又大幅减少了外部 API 调用。
