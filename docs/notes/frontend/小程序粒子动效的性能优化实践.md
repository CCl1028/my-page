# 小程序粒子动效：从「越播越卡」到稳定 60fps

## 一、引言

某天下午收到一个 bug 反馈：「粒子动效第一次播挺流畅的，多点几次就越来越卡，最后整个页面都动不了了。」

这个动效的需求很简单——用户触发某个操作后，屏幕上喷射出一片粒子（心形、星星、气泡之类），从某个角落向对侧飘散，持续约 3 秒后消失。单看需求，不过是 55 个元素做一段位移 + 淡出动画，似乎没什么难度。

但「越播越卡」这五个字，暴露了一个比动画本身更棘手的问题：**资源泄漏**。每播一次，某些东西没有被正确释放，累积到第 5、6 次时，渲染层已经不堪重负。

本文记录了排查和优化这个组件的完整过程，从定位六个独立的性能问题，到最终实现「无限次重复播放、稳定 60fps、零内存增长」的效果。这些优化策略适用于任何小程序/H5 中的粒子动效场景。

---

## 二、问题定位：为什么越播越卡

拿到代码后，逐一排查，发现了六个独立的问题——它们叠加在一起，形成了「一次比一次卡」的复合效应：

| # | 问题 | 为什么导致越来越卡 |
|---|------|-------------------|
| 1 | CSS `animation: ... infinite` | 无限循环动画即使 DOM 移除，渲染层的 GPU 资源也不一定释放 |
| 2 | 停止时数据数组未清空 | 旧数据残留内存，GC 压力逐次递增 |
| 3 | 80 个 `<image>` 组件 | 每个 image 独立解码，缓存累积不释放 |
| 4 | Observer 无防重复 | 快速多次触发导致多份动画叠加 |
| 5 | 每次播放重新生成 DOM | 55 个节点 × 每次 setData ~8KB，渲染层频繁创建/销毁 |
| 6 | `left/right/bottom` 做动画 | 每帧触发布局重排（reflow），CPU 满载 |

前四个导致「泄漏累积」，后两个导致「单次就不够流畅」。接下来逐一击破。

---

## 三、优化策略

### 策略 1：消灭 infinite 动画

这是「越播越卡」的头号元凶。

```css
/* ❌ infinite 动画即使 DOM 移除也可能不释放 GPU 资源 */
animation: breathe 1s ease-in-out infinite;

/* ✅ 有限次数，播完自动停止，资源可被回收 */
animation: breathe 1s ease-in-out 3;  /* 3次 ≈ 3秒，覆盖动效时长 */
```

原理：有限次数动画播完后状态变为 `finished`，浏览器可以安全释放该动画对应的 GPU 纹理和计算器。而 `infinite` 动画永远处于 `running` 状态，渲染层会一直持有资源——即使逻辑层已经把节点"删"了。

### 策略 2：停止时完整清理

原来的 `clearEffect()` 只清了定时器，数据数组还挂在内存里：

```javascript
// ✅ 停止时重置状态，配合 wx:if 移除 DOM 后内存完全释放
stopEffect() {
  this.data._isPlaying = false
  this.clearAllTimers()
  this.setData({ isAnimating: false })
}
```

### 策略 3：view + CSS background 替代 image 组件

```xml
<!-- ❌ 55 个 <image> 组件，每个独立解码同一张图 -->
<image src="https://cdn/particle.png" />

<!-- ✅ CSS background-image，浏览器/小程序只解码一次，全部节点共享 -->
<view class="particle-img"></view>
```

```css
.particle-img {
  background-image: url('https://cdn/particle.png');
  background-size: contain;
}
```

额外收益：setData 不再需要传递 55 次完整 URL 字符串。

### 策略 4：状态锁防重复触发

```javascript
playEffect() {
  if (this.data._isPlaying) return  // 正在播放？忽略
  this.data._isPlaying = true
  // ...
}
```

一行代码杜绝「快速连点导致多份动画叠加」的问题。

### 策略 5：预计算 + DOM 复用（最关键的单项优化）

原来的逻辑是「每次播放都重新生成 55 个随机数据对象 → setData → 渲染层创建 55 个 DOM」。优化思路：**数据只算一次，DOM 常驻不销毁，通过 class 切换驱动动画播放/停止。**

```javascript
// 组件挂载时：生成全部数据（只做一次）
attached() {
  this.setData({
    particles: this.generateParticles(35),
    glowSpots: this.generateGlowSpots(20),
  })
}

// 播放时：只切换一个布尔值
playEffect() {
  this.setData({ isAnimating: false }, () => {
    setTimeout(() => this.setData({ isAnimating: true }), 20)
  })
}
```

```xml
<!-- class 切换驱动动画 -->
<view class="particle {{isAnimating ? 'particle--animating' : ''}}" style="{{item.style}}">
```

| 场景 | 优化前 | 优化后 |
|-----|-------|-------|
| 第 2 次播放 | 计算 55 个对象 + setData 8KB + 创建 55 个 DOM | setData 1 个布尔值 |
| 第 10 次播放 | 同上 + 前 9 次的资源可能泄漏 | 同上，零累积 |

那么问题来了：CSS animation 不支持"重播"，class 一直在那里怎么让动画重新开始？

技巧是 **class 切换 + 20ms 间隔**：先移除动画 class（`isAnimating: false`），等一帧让渲染层识别到动画已移除，再重新添加（`isAnimating: true`），浏览器会把它当作一个全新的动画开始播放。20ms 是确保渲染层完成一次 DOM 更新的安全间隔。

### 策略 6：transform 替代 position（GPU 加速）

这是单次播放流畅度的关键。

```css
/* ❌ left/right/bottom 每帧触发 Layout → Paint → Composite 全链路 */
@keyframes moveX { 0% { right: 0; } 100% { right: 800rpx; } }

/* ✅ translate3d 只触发 Composite，跳过 Layout 和 Paint */
@keyframes particleMove {
  0%   { transform: translate3d(var(--sx), var(--sy), 0) rotate(0deg); }
  100% { transform: translate3d(var(--ex), var(--ey), 0) rotate(var(--rotate-deg)); }
}
```

渲染管线对比：

```
position 动画: Style → Layout → Paint → Composite  (CPU 全程参与，每帧重排)
transform 动画: Style → Composite                   (GPU 直接合成，跳过两个最重的阶段)
```

同时合并原来的 4 个动画（moveX + moveY + spin + fadeOut）为 2 个（move + fadeOut），动画实例从 220 个降至 110 个。

配合 `will-change: transform, opacity` 和 `transform: translateZ(0)` 提前将元素提升为独立合成层，避免动画开始时的首帧卡顿。

## 四、CSS 变量驱动随机轨迹

55 个粒子各有不同的起点、终点、速度、延迟和旋转角度。如果用 JS 每帧计算位置（setInterval + setData），在小程序双线程架构下完全不可行。

解法：**一次性将所有运动参数以 CSS 变量写入 inline style，动画全程由纯 CSS 驱动，零 JS 开销：**

```javascript
particles.push({
  id: i,
  style: `
    width: ${size}rpx; height: ${size}rpx;
    --start-right: ${startRight}rpx;
    --start-bottom: ${startBottom}rpx;
    --move-x: ${moveX}rpx;
    --move-y: ${moveY}rpx;
    --duration: ${duration}s;
    --delay: ${delay}s;
    --rotate-deg: ${rotateDeg}deg;
    --fade-animation: ${fadeAnimation};
  `
})
```

```css
.particle--animating {
  animation:
    particleMove var(--duration) ease-out forwards,
    var(--fade-animation) var(--duration) ease-in forwards;
  animation-delay: var(--delay), var(--delay);
}
```

**淡出时间随机化**：定义 5 套 fadeOut 关键帧（分别在动画进度的 20%/35%/50%/65%/80% 处开始淡出），每个粒子随机分配一套。这样粒子不是齐刷刷消失，而是有层次地逐渐消散，视觉效果更自然。

```css
@keyframes fadeOut1 { 0% { opacity:0 } 10% { opacity:1 } 20% { opacity:1 } 100% { opacity:0 } }
@keyframes fadeOut2 { 0% { opacity:0 } 10% { opacity:1 } 35% { opacity:1 } 100% { opacity:0 } }
@keyframes fadeOut3 { 0% { opacity:0 } 10% { opacity:1 } 50% { opacity:1 } 100% { opacity:0 } }
/* ... */
```

## 五、最终效果

| 指标 | 优化前 | 优化后 |
|-----|-------|-------|
| 重复播放 setData 量 | ~8KB（55 个对象） | 1 个布尔值 |
| 动画实例数/次 | 220 | 110 |
| 渲染管线 | Layout → Paint → Composite | Composite only |
| 内存表现 | 播 5 次后明显增长 | 无限次稳定不变 |
| 帧率 | 第 1 次 50fps → 第 5 次 <30fps | 稳定 60fps |
| 重播延迟 | ~200ms（重建 DOM） | ~20ms（切换 class） |

---

## 六、可复用的结论

1. **永远不要对短时动效使用 `infinite`**——用有限次数覆盖动效时长即可。
2. **粒子轨迹由 CSS 变量 + keyframes 驱动**——JS 只负责一次性生成参数，运行时零开销。
3. **DOM 复用 + class 切换 > 销毁/重建**——重播代价从"创建 N 个节点"降为"改 1 个布尔值"。
4. **只用 transform + opacity 做动画**——跳过 Layout 和 Paint，直接 GPU 合成。
5. **20ms class 切换间隔**是 CSS 动画"重播"的通用 trick。

这些原则不限于小程序，在任何 Web 环境中的粒子/礼花/气泡动效都适用。
