# 小程序全局音频管理器的单例设计

## 背景

小程序里多个页面/组件都要播放音频（环境 BGM、活动音乐、音效等）。如果各自创建 `InnerAudioContext`，会出现：

1. **多音频并行**：两个组件同时播放，用户听到的是混音
2. **资源泄漏**：页面切换时没销毁旧实例，内存越来越大
3. **事件重复触发**：某些机型的 `onPlay` 回调短时间内连续调用 2~3 次，导致状态错乱

解法很直接：**全局单例管理器，确保任意时刻只有一个音频实例存活。**

---

## 核心设计

```javascript
class AudioManager {
  constructor() {
    this.audioContext = null      // 唯一音频实例
    this.currentBgmPath = null   // 当前播放路径
    this.isPlaying = false
    this.isPaused = false
    this.lastPlayEventTime = 0   // onPlay 防抖时间戳
    this.initAudioOption()
  }
}

// 模块级单例：导出实例而非类
export default new AudioManager()
```

---

## 三个关键优化

### 1. 防重复播放

```javascript
play(url, options = {}) {
  // 路径相同 + 正在播放 → 跳过，避免重复创建实例和网络请求
  if (this.isPlaying && this.currentBgmPath === url) return

  this.stop()                         // 先销毁旧实例
  this.createAudioContext(url, options) // 再创建新实例
  this.audioContext.play()
}
```

### 2. onPlay 事件防抖

某些机型（尤其低端 Android）会在极短时间内连续触发 `onPlay`，导致回调执行多次。

```javascript
audio.onPlay(() => {
  const now = Date.now()
  // 500ms 内且已在播放 → 忽略重复事件
  if (now - this.lastPlayEventTime < 500 && this.isPlaying) return

  this.lastPlayEventTime = now
  this.isPlaying = true
  if (this.onPlayCallback) this.onPlayCallback()
})
```

### 3. iOS 静音模式适配

iOS 默认遵循静音开关——用户开了静音，`InnerAudioContext` 就没声音。对于需要强制出声的场景（如冥想引导音乐），需要初始化时设置：

```javascript
initAudioOption() {
  const systemInfo = wx.getSystemInfoSync()
  if (compareVersion(systemInfo.SDKVersion, '2.3.0') >= 0) {
    wx.setInnerAudioOption({ obeyMuteSwitch: false })
  }
}
```

## 完整 API

```javascript
import audioManager from '@/nav_utils/audio-manager'

// 播放（自动停止旧音频）
audioManager.play(url, {
  loop: true,
  onPlay: () => {},
  onStop: () => {},
  onError: (err) => {}
})

// 暂停 / 恢复
audioManager.pause()
audioManager.resume()

// 停止并销毁实例
audioManager.stop()

// 页面卸载时完整清理
audioManager.destroy()

// 查询状态
const { isPlaying, isPaused, currentBgmPath } = audioManager.getStatus()
```

## 跨页面/组件协作模式

```
页面（负责生命周期）
  ├─ onLoad: audioManager.play(主BGM)
  ├─ 子组件A进入: audioManager.play(活动BGM)  ← 自动停掉主BGM
  ├─ 子组件A退出: audioManager.play(主BGM)    ← 恢复
  └─ onUnload: audioManager.destroy()         ← 完整清理
```

**原则**：
- **页面**负责初始化和销毁
- **组件**只负责切换 BGM，不负责销毁（避免影响其他组件）
- 管理器内部保证互斥：`play()` 会先 `stop()` 旧实例

## 资源销毁的防御式编程

`InnerAudioContext` 的 `stop()` 和 `destroy()` 在某些状态下会抛异常（如已停止时再停止）。必须 try-catch 包裹：

```javascript
stop() {
  if (this.audioContext) {
    try { this.audioContext.stop() } catch (e) {}
    this.destroyAudioContext()
  }
}

destroyAudioContext() {
  if (this.audioContext) {
    try { this.audioContext.destroy() } catch (e) {}
    this.audioContext = null
    this.currentBgmPath = null
    this.isPlaying = false
    this.isPaused = false
  }
}
```

## 可复用结论

1. **单例 + 模块导出**。`export default new AudioManager()` 确保全局唯一，多组件共享同一实例。
2. **play 前先 stop**。切换音频时先销毁旧实例，保证内存中只有一个 `InnerAudioContext`。
3. **onPlay 防抖 500ms**。某些机型事件重复触发是已知问题，时间戳防抖是最简单的解法。
4. **iOS 静音开关必须显式处理**。`wx.setInnerAudioOption({ obeyMuteSwitch: false })` 在 SDKVersion ≥ 2.3.0 时可用。
5. **try-catch 包裹所有音频操作**。`stop()`/`destroy()`/`play()` 都可能异常，不能让它阻断业务逻辑。
6. **组件不销毁管理器**。只有页面的 `onUnload` 才调用 `destroy()`，组件销毁时不应干预全局音频状态。
